## Design Patterns

### 1. Serverless-First

- No persistent connections
- Neon serverless driver for DB
- Vercel Edge Functions compatible

### 2. Non-Blocking Operations

- Email sending doesn't block registration response
- Graceful fallbacks if external services fail

### 3. AI-Augmented Content

- Groq generates email content
- Fallback to static templates if AI fails
- Dynamic language based on user preference

### 4. Environment-Driven Configuration

- Drop settings via ENV vars
- No hardcoded values for business logic
- Easy configuration per deployment

## Key Design Decisions

| Decision                      | Rationale                                      |
| ----------------------------- | ---------------------------------------------- |
| Neon over traditional PG      | Serverless, no connection pool management      |
| Groq over OpenAI              | Faster, cheaper for simple text generation     |
| Resend over SendGrid          | Developer-friendly, better Next.js integration |
| No authentication             | MVP simplicity, Season 0 doesn't need it       |
| Cron for welcome emails       | Batch processing, not time-critical            |
| Immediate registration emails | User expects instant confirmation              |

## Security Measures

1. **Environment Variables**: All secrets in ENV, never in code
2. **Cron Protection**: `CRON_SECRET` header validation
3. **Email Validation**: Strict alphanumeric before @
4. **SQL Injection**: Parameterized queries via Neon tagged templates
5. **HTTPS Only**: Enforced by Vercel
